## 简介

查询任务列表接口用于获取 CodeReviewX 系统中所有 Review 任务的分页列表。该接口支持分页查询，允许客户端指定页码和每页大小来控制返回的任务数量。

## 接口概述

- **HTTP 方法**: GET
- **端点**: `/api/review-tasks`
- **功能**: 获取 Review 任务列表
- **当前状态**: 计划中（Round 01 仅为计划设计，未实现）

## 查询参数

| 参数名称 | 类型 | 是否必填 | 默认值 | 说明 |
|---------|------|----------|--------|------|
| `page` | integer | 否 | 0 | 页码，从 0 开始计数 |
| `size` | integer | 否 | 20 | 每页显示的任务数量 |

### 参数限制

- `page`: 必须为非负整数
- `size`: 必须为正整数，建议范围 1-100
- 当 `page` 超出实际页数范围时，将返回空数组

## 响应格式

接口返回一个包含任务列表和总数的 JSON 对象：

```json
{
  "items": [
    {
      "taskId": 1,
      "repoUrl": "https://github.com/owner/repo",
      "prNumber": 12,
      "status": "SUCCESS",
      "riskLevel": "MEDIUM",
      "createdAt": "2026-06-19T10:00:00"
    }
  ],
  "total": 1
}
```

## 字段详细说明

### 响应对象字段

| 字段名称 | 类型 | 必填 | 说明 |
|---------|------|------|------|
| `items` | array | 是 | 任务列表数组 |
| `total` | integer | 是 | 符合查询条件的总任务数 |

### items 数组中的任务信息

| 字段名称 | 类型 | 必填 | 说明 |
|---------|------|------|------|
| `taskId` | long | 是 | 任务唯一标识符 |
| `repoUrl` | string | 是 | GitHub 仓库地址，格式：`https://github.com/{owner}/{repo}` |
| `prNumber` | integer | 是 | Pull Request 编号，必须为正整数 |
| `status` | string | 是 | 任务状态，取值范围：`PENDING`、`RUNNING`、`SUCCESS`、`FAILED` |
| `riskLevel` | string | 否 | 风险等级，取值范围：`LOW`、`MEDIUM`、`HIGH`，未完成时为 null |
| `createdAt` | string | 是 | 任务创建时间，ISO 8601 格式 |

## 状态说明

### 任务状态枚举

| 状态值 | 含义 | 描述 |
|-------|------|------|
| `PENDING` | 待执行 | 任务已创建，尚未开始执行 |
| `RUNNING` | 执行中 | 任务正在执行分析过程 |
| `SUCCESS` | 执行成功 | 任务分析完成，生成了 Review 结果 |
| `FAILED` | 执行失败 | 任务执行过程中发生错误 |

### 风险等级枚举

| 等级值 | 含义 | 描述 |
|-------|------|------|
| `LOW` | 低风险 | 代码审查中发现的问题风险较低 |
| `MEDIUM` | 中风险 | 代码审查中发现的问题风险中等 |
| `HIGH` | 高风险 | 代码审查中发现的问题风险较高 |
| null | 未完成 | 任务尚未完成分析，因此没有风险评估 |

## 错误处理

### 错误响应格式

当请求参数无效或发生服务器错误时，接口返回统一的错误响应格式：

```json
{
  "code": "ERROR_CODE",
  "message": "Human readable error message",
  "details": null
}
```

### 错误码定义

| 错误码 | HTTP 状态 | 场景 |
|-------|-----------|------|
| `INVALID_REQUEST` | 400 | 请求参数错误或校验失败 |
| `TASK_NOT_FOUND` | 404 | 任务不存在 |
| `AI_SERVICE_ERROR` | 502 | ai-service 调用失败 |
| `GITHUB_FETCH_FAILED` | 502 | GitHub 数据获取失败 |
| `DATABASE_ERROR` | 500 | 数据库操作失败 |
| `INTERNAL_ERROR` | 500 | 未知系统错误 |

## 最佳实践

### 分页查询建议

1. **合理设置 size 值**：建议使用 20-50 之间的数值，避免过大导致响应缓慢
2. **处理空结果**：当 total 为 0 时，应向用户显示"暂无任务"的提示
3. **边界情况处理**：当 page 超出范围时，应优雅地处理空数组返回

### 前端集成建议

1. **加载状态**：在请求发送时显示加载动画
2. **错误处理**：对各种错误码进行友好的用户提示
3. **缓存策略**：对于相同查询参数的结果可以考虑适当的缓存

### 后续实现注意事项

1. **数据一致性**：确保 status 和 riskLevel 字段的更新时机
2. **性能优化**：对大量数据的查询进行索引优化
3. **安全性**：对 repoUrl 进行必要的 URL 校验和过滤
